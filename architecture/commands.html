Mover un rectángulo
una operación interactiva y un comando puro

Mover tiene dos caras: una operación que conduce el gesto —arranca el picking, te deja recoger un elemento válido, lo dibuja moviéndose y, con un segundo click sobre una caja, decide dónde guardarlo— y un comando puro que registra el cambio para poder deshacerlo. Aquí están los elementos, por qué se separan, cómo se relacionan, la secuencia del gesto y cómo se gestiona el undo. Cada elemento tiene su color fijo.

01

El reparto · quién es quién

Prisma estructural — qué objetos existen

La pieza clave: MoveOperation (el conductor del gesto) y MoveCommand (el cambio undoable) son dos objetos distintos. El Picker es el servicio que espera cada click, y las cajas son los destinos donde se guarda el rectángulo.

operación · conductor interactivo
MoveOperation
Conduce el gesto y mantiene el estado (EsperandoPick → Moviendo → coloca). Arranca el picking y, al final, produce el comando. Cancelable, ocurre una vez, no va a la pila de undo.
recibe: IPicker, IPreview, CommandBus
input
Picker
Espera el siguiente click y dice qué hay debajo, filtrado (movible / caja / punto). Reusable por todas las operaciones — la operación lo llama, no al revés.
implementa: IPicker · pick(filtro) → Hit
elemento
Rectangle
Lo que se mueve: un id + colocación (posición y padre). Puede acabar guardado dentro de una caja.
lo posee: Scene
destino · contenedor
Box · caja
Sitio donde guardar el rectángulo. El segundo click sobre una caja determina cómo colocarlo: encaja en su hueco.
lo posee: Scene · puede contener un Rectangle
command · cambio puro
MoveCommand
El cambio undoable que la operación produce: (elemento, destino). Guarda el estado anterior para deshacer. Sin UI, sin esperas: se re-ejecuta tal cual en redo.
recibe: ISceneEditor (y ITransformService)
invoker
CommandDispatcher
Ejecuta el comando y mantiene las pilas de undo/redo. Los llama de forma uniforme, sin conocer las dependencias de cada uno.
conoce a: ICommand
documento
Scene · TransformService
Posee rectángulos y cajas; mueve y re-parenta (mete el rect en la caja), sirve el hit-test al picker y emite sceneChanged.
implementa: ITransformService, ISceneEditor
observador
RenderManager
Observa la escena y redibuja, incluido el estado “moviéndose” mientras el rectángulo está en el aire.
conoce a: Scene (como observador)
02

Operación vs comando · y qué recibe cada uno

La decisión de fondo — dos ciclos de vida, no uno

La operación centraliza la secuencia del gesto (por eso escala cuando la operación tiene más pasos) y arranca el picking llamando al Picker. Pero lo que va a la pila de undo no es ella: es el comando puro que produce al final.

MoveOperation

ciclo de vidatransitorio · cancelable · una vez
espera clickssí — arranca y conduce el picking
recibe (concreto)IPicker · IPreview · CommandBus
¿pila de undo?no
MoveCommand
cambio puro
ciclo de vidasíncrono · se repite en redo · serializable
espera clicksnunca — recibe elemento y destino ya resueltos
recibe (concreto)ISceneEditor (· ITransformService)
¿pila de undo?sí

Por qué separarlos: si el objeto de la pila fuera el que espera clicks, el redo volvería a pedirte que hagas click (debe ser silencioso), no podrías scriptear un batch, y no sería serializable. Y separados, cada uno recibe solo sus interfaces: interacción para la operación, mutación para el comando.

Nada de contexto-cajón. En vez de pasar un contexto con todos los managers, inyecta a cada uno las interfaces concretas que usa (por constructor). Así execute()/undo() no necesitan contexto y el dispatcher los llama uniforme. Un “contexto” pequeño solo se justifica para lo ambiental (documento activo, transacción), nunca como bolsa de servicios. Y la ergonomía del cableado se resuelve con una factory en el composition root, no con una bolsa en tiempo de ejecución.

// La OPERACIÓN: conduce el gesto. Recibe SOLO servicios de interacción.
class MoveOperation {
    IPicker& picker_;  IPreview& preview_;  CommandBus& bus_;  MoveFactory make_;
public:
    Task run() {                                    // arranca al elegir la opción
        auto rect = co_await picker_.pick(Filter::Movable);       // 1er click
        if (!rect) co_return;                                     // Esc: nada que deshacer
        preview_.begin(*rect);                                    // "moviéndose"
        auto target = co_await picker_.pick(Filter::BoxOrPoint);  // 2º click
        preview_.clear();
        if (!target) co_return;
        bus_.execute( make_(*rect, *target) );      // la factory crea el comando con SUS deps
    }
};

// El COMANDO: cambio puro. Recibe SOLO servicios de mutación. Sin contexto.
class MoveCommand : public ICommand {
    ISceneEditor& scene_;
    EntityId what_;  Placement to_;
    std::optional<Placement> from_;         // estado anterior (ver §05)
public:
    MoveCommand(ISceneEditor& s, EntityId e, Placement to) : scene_(s), what_(e), to_(to) {}
    void execute() override { if(!from_) from_ = scene_.placementOf(what_); scene_.place(what_, to_); }
    void undo()    override { scene_.place(what_, *from_); }
};

// El dispatcher es genérico: no conoce las deps de nadie.
void CommandBus::execute(std::unique_ptr<ICommand> c) { c->execute(); undo_.push(std::move(c)); redo_.clear(); }
El co_await hace que la espera del click no bloquee el hilo de UI (corrutina). Sin corrutinas, lo mismo con callbacks o una máquina de estados avanzada por eventos.
03

El mapa de relaciones

Prisma de dependencias — quién apunta a quién

Flechas sólidas = “usa / depende de”; discontinuas = “implementa” u “observa”. Fíjate en los dos conjuntos de interfaces: la operación usa IPicker; el comando usa ITransformService·ISceneEditor. Ninguno recibe un contexto con todo.

El Picker sirve los dos clicks (usa el hit-test de la escena por dentro); el comando recoloca vía la escena, que contiene los rectángulos y las cajas.

05

Cómo se deshace

Prisma del undo — guardar el estado, no recomputar a ciegas

Dos estrategias. Por defecto uso la primera: el comando guarda el estado anterior de lo que pisa (memento dirigido) y lo restaura. El inverso por lógica solo cuando la operación es exacta y sin efectos colaterales.

Memento dirigido
recomendado

Guarda { posición, padre } previos y los vuelve a escribir. Correcto aunque haya efectos colaterales, y sin deriva de floats (restaura el valor absoluto anterior).

Inverso por lógica
casos simples

Recomputa la marcha atrás (-delta). Solo para operaciones exacta y baratamente invertibles, sin efectos colaterales. Ojo: casi siempre acabas guardando el origen igualmente.

void execute() override {
    if (!from_) from_ = scene_.placementOf(what_);   // MEMENTO: guardo lo que voy a pisar (1er execute)
    scene_.place(what_, to_);                         // posición + re-parent en la caja
}
void undo() override {
    scene_.place(what_, *from_);                      // restauro ese estado, exacto
}
// redo = execute() otra vez: from_ ya está, solo reaplica to_

Detalle: se captura en execute(), no en el constructor (el estado a restaurar es el del momento en que se ejecuta), y el redo no vuelve a capturar.

Para operaciones futuras complejas, no escribas un undo() gigante a mano:

Composite
Macro de comandos
La operación compleja es un MacroCommand de sub-comandos pequeños, cada uno con su propio undo. Deshacer = deshacer los hijos en orden inverso. Un solo paso de undo para el usuario.
Journal · transacción
Diario de cambios
El modelo registra cada mutación durante execute() y el sistema de undo invierte el diario automáticamente. Nada de undo a mano; a cambio, el modelo debe soportar change-tracking.
